# 动态Cookie和Token实施计划

## 目标
1. 实现动态cookie加载（使用session）
2. 实现动态token获取（通过正则匹配HTML中的radon = `xxx`）
3. 为每个不同的请求单独创建数据模型，而不是混用

## 当前状态分析
- Token硬编码在`lib/constants/api_constants.dart`中：`static const String token = '98ae9979efca566d';`
- Cookie硬编码在同一文件中，包含特定的session信息
- 所有请求使用统一的`CalculationRequest`模型

## 实施步骤

### 第一步：创建动态Token和Cookie服务
1. 创建新服务 `lib/services/dynamic_auth_service.dart`
2. 实现获取mathdf.com主页的功能
3. 使用正则表达式匹配HTML中的 `radon = \`xxx\`` 提取token
4. 维护cookie会话，自动更新从服务器响应中的Set-Cookie头
5. 提供方法获取当前token和cookie

### 第二步：修改API服务以使用动态认证
1. 修改 `lib/services/api_service.dart`
2. 移除对硬编码token的依赖
3. 使用动态认证服务获取当前token
4. 使用动态认证服务管理的cookie而不是硬编码的cookie
5. 确保每个请求使用更新的认证信息

### 第三步：为不同请求类型创建专门的数据模型
经过分析，当前的`CalculationRequest`模型已经足够灵活，可以通过不同的工厂方法和参数处理不同的计算类型。不需要创建单独的数据模型，而是确保API服务正确地使用不同的参数组合。

### 第四步：测试和验证
1. 确保动态token获取正常工作
2. 验证cookie会话正确维护
3. 测试所有计算类型的功能
4. 确保没有向后兼容性问题

## 技术细节

### 动态认证服务
- 使用Dio获取 https://mathdf.com 主页
- 正则表达式: `radon = \`([^`]*)\``
- 维护cookie jar以自动处理cookie
- 提供获取当前token和cookie header的方法
- 添加错误处理和重试机制
- 考虑添加缓存机制以减少不必要的请求

### API服务修改
- 注入动态认证服务（通过构造函数）
- 在发送请求前获取最新token和cookie
- 移除BaseOptions中的硬编码cookie
- 在每个请求中动态设置headers中的cookie
- 更新加密密钥的构造方式，使用动态token
- 确保错误处理包括认证失败的情况

### API常量修改
- 移除硬编码的token常量
- 移除硬编码的cookie常量
- 保留其他API常量（baseUrl、端点、超时等）

## 风险和注意事项
- 确保网络请求不会造成性能问题：考虑在App启动时预先获取认证信息
- 考虑缓存token以减少不必要的请求：token在一定时间内可能保持有效
- 处理token获取失败的情况：提供后备机制或明确的错误提示
- 确保cookie在不同请求之间正确隔离：使用共享的CookieJar应该能自动处理
- 考虑到不同计算类型可能需要不同的参数处理：当前模型通过工厂方法已经支持

## 实施顺序
1. 创建动态认证服务
2. 修改API服务使用动态认证
3. 修改API常量移除硬编码值
4. 测试所有功能
